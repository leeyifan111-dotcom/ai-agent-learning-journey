## 💡 今日最大领悟（值钱程度 A+）

### 🔥 单进程内嵌工具，HITL 是"假的"

按 y 后程序仍然崩——因为工具代码已被 Python 加载到内存，**人不可能在程序卡住时真改它**。这暴露了一个根本问题：

> **HITL 的边界 = 工具的部署边界**
>
> - 工具跟 agent 在同一进程 → HITL 只是 `input()` 摆设，按 y 无意义
> - 工具独立部署（HTTP/MCP server）→ HITL 才是真护栏

这一步**自己推导出了 #027 MCP 存在的根本理由**——不是因为协议好看，是因为**工具不解耦 HITL 就是假的，演示用都演示不下去**。

### 📊 三种工具部署架构 vs HITL 真伪

| 架构                        | 工具修复方式                             | HITL 是否真生效 |
| --------------------------- | ---------------------------------------- | --------------- |
| 工具内嵌（tiny-agent-loop） | 必须重启进程                             | ❌ 假           |
| HTTP API                    | 改远程服务并重启**那个服务**，agent 不动 | ✅ 真           |
| MCP server                  | 改 MCP server 并重启它，agent 自动重连   | ✅✅ 规范化的真 |

→ 解耦不只是为了"工具复用"（MCP 文档讲的角度），更是为了**让运维介入有可操作空间**。

## ❓ 困惑 / 待消化

### Q1：y 分支为什么会崩？

**答**：当时 `safe_dispatch` 写的是：

```python
except FatalError as e:
    choice = input(...)
    if choice == "y":
        return TOOL_REGISTRY[name](**args)   # ← 裸调，没 try
```

按 y 后再次调用同一工具，火星还是火星 → 又抛 FatalError → **已在 except 块内，外层没人接** → 程序崩栈。

**修法**：y 分支也包 try，捕住二次失败：

```python
if choice == "y":
    try:
        return TOOL_REGISTRY[name](**args)
    except Exception as retry_err:
        return f"error 工具 {name} 人工修复后仍失败: {retry_err}"
```

### Q2：为什么按 y 一定会再次失败？

**答**：Python 启动时把模块代码加载进内存。程序卡在 `input()` 等输入时，你**没有任何办法**改已加载的字节码：

- 编辑器里改了 `.py` 文件 → 内存里那份没变
- 改了 JSON 配置 → 工具里没读配置，照旧报错
- 唯一的"修复"是**重启进程**——但重启就丢失了对话上下文

→ 这就是 Q1 引出的根本问题：**单进程内嵌的"修复"是个悖论**。

### Q3：那 HITL 在 demo 场景下还有什么价值？

**答**：仍然有，但价值降级了：

| 价值层级                    | demo 内嵌           | 解耦后（HTTP/MCP） |
| --------------------------- | ------------------- | ------------------ |
| **强制停下来等人**          | ✅ 仍有效           | ✅                 |
| **让人接管最终决策（y/n）** | ✅ 仍有效（n 分支） | ✅                 |
| **让人真的修工具再继续**    | ❌ 假的             | ✅                 |

→ demo 至少证明了"程序在关键路径必须停"，不是完全没用。但要做真 HITL 必须解耦。
