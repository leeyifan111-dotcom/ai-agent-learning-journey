## 📋 六道任务

### Task 1 · TOOL_REGISTRY dict 重构派发表 · 5 分钟

**复习钩子**：[#021 Function Calling](daily/2026-06-04/021-function-calling-basics.md) — 派发表机制

**现状（v0.2）**：

```python
if name == "get_weather":
    result = get_weather(**args)
elif name == "calculator":
    result = calculator(**args)
elif name == "get_weather_batch":
    result = get_weather_batch(**args)
```

**改成**：

```python
TOOL_REGISTRY = {
    "get_weather": get_weather,
    "calculator": calculator,
    "get_weather_batch": get_weather_batch,
}
result = TOOL_REGISTRY[name](**args) if name in TOOL_REGISTRY else f"未知工具: {name}"
```

**完成判定**：

- [ ] if/elif 派发逻辑全部删除
- [ ] 加新工具时只需在 `TOOL_REGISTRY` 加一行 + `tools` schema 加一段

**复习自问**：「为什么 Day 1 加 `get_weather_batch` 时报『未知工具』？dict 派发表如何防御这个 bug？」

### Task 2 · 加 system prompt（生产六件套全员） · 20 分钟

**复习钩子**：[#059 System Prompt](daily/2026-06-09/059-system-prompt-principles.md) + [#062 Agentic Prompting](daily/2026-06-09/062-agentic-prompting.md)

**做法**：在 `messages` 初始化时塞一段 system 消息：

```python
SYSTEM_PROMPT = """你是一个数据查询助手，在一个 Agent 循环里工作。

# 工具使用规则
- 实时信息（天气/股价/新闻）必须调用对应工具，禁止用内置知识回答
- 工具失败 → 立即停止，禁止换花样重试

# 思维流程（每步必须先 thought 再 action）
- thought：我下一步要做什么？为什么？
- action：调哪个工具，参数是什么？或者：FINAL_ANSWER

# 不确定性处理
- 不知道就说"我不知道"，禁止编造

# 终止信号
- 完成：输出 `FINAL_ANSWER: <答案>`
- 失败：输出 `CANNOT_COMPLETE: <原因>`

# 关键约束（再说一遍，防止 Lost in the Middle）
- 工具失败必须停手
- 不知道必须说
- 高危操作必须问用户
"""

messages = [
    {"role": "system", "content": SYSTEM_PROMPT},
    {"role": "user", "content": user_input},
]
```

**完成判定**：

- [ ] system prompt 涵盖六件套：身份/工具规则/思维触发/不确定性/格式/结尾重申
- [ ] 关键约束在开头+结尾各出现一次（对抗 Lost in the Middle）

**复习自问**：「为什么关键约束要写两遍？」「What not How 原则在我这段 prompt 里体现在哪？」

### Task 3 · 错误分层 dispatch（可恢复 vs 不可恢复） · 20 分钟

**复习钩子**：[#006 Agent Loop 与错误恢复](daily/2026-06-04/006-agent-loop-and-error-recovery.md)

**做法**：定义两类异常 + 在 dispatch 里分别处理：

```python
class RecoverableError(Exception): pass     # 网络抖动、超时 → 重试
class FatalError(Exception): pass            # 权限错误、工具异常 → 立即停

def safe_dispatch(name, args, max_retry=2):
    for attempt in range(max_retry + 1):
        try:
            return TOOL_REGISTRY[name](**args)
        except RecoverableError as e:
            if attempt == max_retry:
                return f"工具 {name} 重试 {max_retry} 次仍失败: {e}"
            continue
        except FatalError as e:
            return f"工具 {name} 不可恢复错误: {e}"
        except Exception as e:
            return f"工具 {name} 未知错误: {e}"
```

**完成判定**：

- [ ] dispatch 包了 try/except，明确区分可恢复 vs 不可恢复
- [ ] 至少一个工具（比如 `get_weather`）模拟抛 `RecoverableError` 触发重试逻辑

**复习自问**：「为什么可恢复错误也要有重试上限？没有上限会怎样？」（对应 #006 的硬上限）

### Task 4 · detect_loop（集合去重检测） · 15 分钟

**复习钩子**：[#006 Q1 detect_loop](daily/2026-06-04/README.md#q1detect_loop-这段代码看不懂来自-006) — 你 L2 复习时完全忘了的那段

**做法**：

```python
def detect_loop(action_history: list[str], window: int = 3) -> bool:
    """检测最近 N 步是否重复操作"""
    recent = action_history[-window:]
    return len(recent) == window and len(set(recent)) < len(recent)

# 在主循环里：
action_history = []
for turn in range(MAX_TURNS):
    resp = client.chat.completions.create(...)
    msg = resp.choices[0].message

    if msg.tool_calls:
        for call in msg.tool_calls:
            action_signature = f"{call.function.name}({call.function.arguments})"
            action_history.append(action_signature)

            if detect_loop(action_history):
                print("⚠️ 检测到循环，强制终止")
                return "循环检测：模型在反复尝试同一操作，已强制终止"

            # ... 正常 dispatch
```

**完成判定**：

- [ ] `detect_loop` 函数实现 + 单元测试两个 case（`["A","B","B"]` → True；`["A","B","C"]` → False）
- [ ] 主循环里每次 tool_call 后都查一次

**复习自问**：「`len(set(recent)) < len(recent)` 这一行为什么能检测重复？为什么 `==` 不行？」

### Task 5 · JSON Mode 强制 {"thought","action"} · 25 分钟

**复习钩子**：[#061 结构化输出](daily/2026-06-09/061-structured-output.md) + [#062 Agentic Prompting](daily/2026-06-09/062-agentic-prompting.md) — 软硬约束双轨

**做法**：在两个关键决策点用 JSON Mode 强制结构：

```python
# 在某些工具失败后，让 LLM 显式表态是 retry 还是 stop
def llm_reflect(error_msg: str) -> dict:
    resp = client.chat.completions.create(
        model="deepseek-chat",
        messages=[
            {"role": "system", "content": "工具失败了，请反思后输出决策"},
            {"role": "user", "content": f"错误: {error_msg}\n输出 JSON: {{\"reflection\": \"...\", \"decision\": \"retry\" | \"stop\"}}"},
        ],
        response_format={"type": "json_object"},
    )
    return json.loads(resp.choices[0].message.content)
```

**完成判定**：

- [ ] 至少一处用上 `response_format={"type":"json_object"}`
- [ ] LLM 在错误恢复时被强制走 `reflection → decision (retry/stop)` 结构

**复习自问**：「软约束（Agentic prompt 教 thought→action）+ 硬约束（JSON Mode 保格式）为什么必须配套？只有软约束会怎样？」

### Task 6 · HITL：工具失败 input() 等人审批 · 10 分钟

**复习钩子**：[#003 Agent 架构](daily/2026-06-08/003-agent-architecture-patterns.md)（通用调优 HITL） + [#007 Workflow vs Agent](daily/2026-06-08/007-workflow-vs-agent.md)（Agentic Workflow）

**做法**：在 `safe_dispatch` 的 `FatalError` 分支加 input() 卡点：

```python
except FatalError as e:
    print(f"⚠️ 工具 {name} 不可恢复错误: {e}")
    choice = input("y = 我已人工修复，请重试；n = 终止任务: ").strip().lower()
    if choice == "y":
        return TOOL_REGISTRY[name](**args)
    return "用户已终止任务"
```

**完成判定**：

- [ ] FatalError 时程序卡住等输入，不再无声失败
- [ ] 用户可以中断后修复工具/参数，再继续

**复习自问**：「为什么 `input()` 比 system prompt 写'高危操作要确认'更可靠？」（对应 #059 软硬约束讨论）
